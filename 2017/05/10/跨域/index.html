前端跨域解决方案

发表于 2017-05-10 | 分类于 前端 | 阅读次数

跨域是我们前端开发人员经常要遇到的问题，也是一个经典问题，总结一下什么是跨域以及自己常用的一些解决方案。

什么是跨域

限制跨域是浏览器的行为，而不是JS的行为，由于安全问题，浏览器有一个”同源策略”，只要协议、域名、端口有任何一个不同，都被当作是不同的域，之间的请求就不会成功。常见的报错为 Failed to load https://www.baidu.com/: No 'Access-Control-Allow-Origin' header is present on the requested resource. Origin 'null' is therefore not allowed access.

JSONP跨域

浏览器对XHR做了同源策略，但并没有将这种方式延续到script上（其实还有iframe，img等），从而可以利用script标签的src属性，动态script标签技术来做到跨域请求的作用。
具体方案为前端事先定义一个处理数据的回调函数，例如叫callback，请求数据的时候后台把数据以JSON格式放到函数中返回，例如callback({"msg":"this is data"})，这样前端一拿到数据就会当做JS解析，跨域就完成了。
JSONP需要后台协助操作，而且请求方式只能为GET。

CORS跨域

CORS全称是”跨域资源共享”(Cross-origin resource sharing)，是一种通过前后端http header配置来进行跨域的一种方式，后台在响应头中设置Access-Control-Allow-Origin: 你的域名，或者Access-Control-Allow-Origin:*（表示允许任何域名访问），前端请求就不会有问题了。
CORS同样需要后台支持。

禁用浏览器安全策略跨域

既然跨域是浏览器的限制，那么取消这层限制就可以了，这种方法适合前端开发本地调试接口的情况，不需要后台做改动。
常用的在Chrome启动时加一个参数：--disable-web-security --user-data-dir，这样浏览器启动后就不会有同源策略的限制了。

代理请求跨域

本人最常用的方案，本地搭建一个服务器(Node.js 的Express/Koa)用来服务前端静态文件，所有接口调用本地的，然后经过本地服务器转发给后台去。这样还有一个好处是可以自己在发请求的时候做一些处理，后台需要什么东西都可以自己设置，比如伪造Referer请求头反防盗链，设置cookie模拟登陆等等。。
